# pose_viewer 前端改造方案

## 1. 目标

本次改造目标有四个：

1. 移除首页 `hero` 区块，不再保留展示型大标题和重复状态卡片。
2. 重新评估 `panel` 内容，只保留练习流程必须的信息，并改造成类似 App 的页面流，不再使用侧边栏。
3. 移除 `stage-shell` 内的 `overlay-card overlay-top collapsed`、`overlay-card overlay-bottom collapsed`、`stage-hud`。
4. 评估并支持手机端的 `stage` 沉浸式全屏显示，但在进入全屏前，`panel` 中的内容必须先在页面流中完成。

## 2. 当前实现概况

当前真实运行入口是 `pose_viewer.html`，不是 `App.jsx`。页面结构可概括为三段：

- `hero`：标题、说明文案、4 个顶部指标
- `panel`：动作选择、摄像头控制、节奏调节、状态面板
- `stage-shell`：视频舞台、JSON 调试卡、HUD、图例、一键开始按钮

当前问题主要有：

- 信息重复：`hero`、`panel`、`stage-hud`、`overlay-card` 都在展示状态。
- 结构耦合：一个 `updateStatusUI()` 同时驱动多个展示区，导致后续改造成本高。
- 布局过于桌面化：`panel + stage` 的双列结构偏工作台，不像移动端 App。
- 舞台内信息过载：调试信息、评分 HUD、控制按钮都压在 `stage` 上。
- 手机端虽然是 9:16 竖屏舞台，但还没有真正意义上的沉浸式全屏模式。

## 3. Hero 处理方案

### 3.1 直接移除

`hero` 建议整体删除，不做保留或变体迁移。

原因：

- `hero` 的标题和说明属于一次性介绍，不是练习流程中的必要信息。
- `hero` 中 4 个指标与 `panel` / `stage-hud` 展示的状态重复。
- 在移动端上，`hero` 会显著拉长首屏，挤压真正有用的交互区域。

### 3.2 替代方式

保留一条更轻量的页面级说明即可，放到准备页顶部：

- 页面标题：`动作跟练`
- 辅助说明：`选择标准动作后即可打开摄像头开始练习`

不再保留指标卡片式信息区。

## 4. Panel 内容评估

当前 `panel` 可以拆成四类能力：

### 4.1 必须保留

- 标准动作选择
- JSON 上传
- 示例动作快捷入口
- 打开摄像头
- 关闭摄像头
- 开始 / 暂停播放
- 镜像切换
- 播放速度调节
- 当前核心状态
- 当前总分 / 结果

### 4.2 应保留但降级

- 左臂最终分
- 右臂最终分
- 动作结论
- 拍点命中率
- 拍点击分
- 下一拍提示

这些内容不需要始终占据主操作区，应放到练习页底部信息区或结果页。

### 4.3 仅高级模式保留

- 当前目标帧
- 当前时间
- 评分体系
- 节奏分析原始状态
- 拍点统计
- 拖动定位某一帧

这些更偏调试或精细控制，不应作为主路径默认显示。

### 4.4 直接移除出主界面

- JSON 帧摘要
- 调试 HUD 文本

这两块可以保留为开发调试能力，但不能继续留在生产界面上。建议改为：

- 默认隐藏
- 仅在 `?debug=1` 或开发模式下显示

## 5. 新的信息架构

不再使用 `aside.panel + section.stage-shell` 的工作台结构，改成单页 App 流程。

建议拆成 3 个页面态：

1. 准备页 `setup-view`
2. 练习页 `practice-view`
3. 结果页 `result-view`

### 5.1 准备页

职责：

- 选择标准动作
- 上传 JSON
- 打开摄像头
- 设置镜像
- 设置播放速度
- 确认练习已准备完成

页面内容建议：

- 顶部标题和简短说明
- 动作选择卡片区
- 摄像头控制区
- 练习设置区
- 主按钮：`开始练习`

### 5.2 练习页

职责：

- 只承载练习过程
- 舞台是页面核心
- 所有必要反馈在页面流内提供，不叠加到视频画面内部

页面内容建议：

- 顶部轻量导航：返回准备页 / 退出练习
- 中间 `stage`
- 底部 `practice-dock`

`practice-dock` 中只保留核心信息：

- 当前状态
- 当前总分
- 左右臂摘要
- 下一拍提示
- 播放控制
- 进入全屏按钮

### 5.3 结果页

职责：

- 展示一轮练习后的结果结算
- 给出复盘信息和重新开始入口

页面内容建议：

- 过程均分
- 最终结论
- 左臂 / 右臂得分
- 拍点命中率
- 拍点击分
- 重新练习
- 返回准备页

## 6. Stage-Shell 改造方案

### 6.1 移除项

以下内容从 `stage-shell` 中删除：

- `overlay-card overlay-top collapsed`
- `overlay-card overlay-bottom collapsed`
- `stage-hud`

删除原因：

- 它们会遮挡视频主体。
- 它们与页面其它信息区重复。
- 在移动端上会直接压缩舞台可视面积。
- 练习态应突出动作画面，不应堆叠太多悬浮信息。

### 6.2 保留项

`stage` 内仅保留：

- `video`
- `canvas`
- 可选的最少量顶部控制按钮

建议 `stage` 内最多保留两类元素：

- 右上角一个极简按钮组：返回、全屏
- 必要的系统级提示，例如权限失败或摄像头不可用

### 6.3 图例处理

当前 `legend` 可保留，但不要继续放成独立卡片式尾部区域。建议改成：

- 练习页底部信息区中的一行简化图例
- 或者放入“帮助说明”抽屉中

默认主流程里不需要持续强调三种颜色图例。

## 7. 手机端全屏评估

## 7.1 结论

`stage` 可以支持手机端近似原生 App 的全屏练习模式。

当前代码已经具备两个基础条件：

- 摄像头请求就是 9:16 竖屏
- `canvas` 会根据实际渲染尺寸自动重算

这意味着算法和绘制层基本无需重写，主要工作在布局层。

## 7.2 当前阻碍

当前还不能真正沉浸式全屏，原因在于：

- 外层 `.shell` 有最大宽度与外边距
- `stage-shell` 仍是卡片容器
- `stage` 高度被限制在 `min(78vh, 820px)`
- 舞台内存在 HUD、overlay、legend 等非必要元素

## 7.3 建议方案

采用两段式体验：

### 页面态

先在页面内完成所有 `panel` 内容：

- 选择动作
- 打开摄像头
- 设置镜像和速度
- 确认开始练习

### 沉浸态

用户点 `进入全屏练习` 后，切换到移动端沉浸式舞台：

- `stage-shell` 脱离普通卡片样式
- `stage` 占满视口
- 底部信息区压缩为一条或一块底部 dock

## 7.4 技术实现建议

移动端全屏建议优先使用 CSS 视口模式，而不是把主方案完全建立在浏览器 Fullscreen API 上。

建议：

- `meta viewport` 增加 `viewport-fit=cover`
- 使用 `100dvh` 作为沉浸式舞台高度
- 使用 `env(safe-area-inset-top)` / `env(safe-area-inset-bottom)` 处理刘海屏与底部安全区
- 保留 Fullscreen API 作为增强能力，而不是主依赖

## 8. 交互流程改造

### 8.1 新主流程

1. 用户进入准备页
2. 选择标准动作或上传 JSON
3. 打开摄像头
4. 设置镜像与播放速度
5. 点击 `开始练习`
6. 进入练习页
7. 在练习页内点击 `进入全屏`
8. 完成练习后进入结果页

### 8.2 全屏前必须在页面内完成的内容

这是本次改造的关键限制，必须明确：

在进入沉浸式练习前，所有原 `panel` 的主操作都必须已经在页面流中完成，不能把核心配置留给全屏态。

也就是说，全屏态只做两件事：

- 看舞台
- 跟练

不承担复杂配置职责。

## 9. 状态展示重组方案

当前 `updateStatusUI()` 同时给多个区域写数据，后续应拆分为更明确的渲染职责。

建议拆成：

- `renderSetupView()`
- `renderPracticeView()`
- `renderPracticeDock()`
- `renderResultView()`
- `renderDebugPanel()`

这样可以避免以下问题：

- 一个状态改动需要同步维护多个重复 DOM
- 删除 `hero` 或 `stage-hud` 时需要大面积回收逻辑
- 移动端和桌面端难以走不同展示策略

## 10. 推荐 DOM 重构方向

建议重构为如下结构：

```html
<div class="app-shell">
  <header class="app-nav"></header>
  <main class="app-main">
    <section class="view setup-view"></section>
    <section class="view practice-view"></section>
    <section class="view result-view"></section>
  </main>
</div>
```

其中练习页内部：

```html
<section class="practice-view">
  <div class="practice-stage-shell">
    <div class="stage"></div>
  </div>
  <div class="practice-dock"></div>
</section>
```

沉浸式模式只在练习页上增加状态类：

```html
<section class="practice-view is-immersive"></section>
```

不建议再新增一套独立页面，避免状态同步复杂化。

## 11. 实施顺序

建议按以下顺序落地，避免一次性改太大：

### 第一步：删除重复展示

- 删除 `hero`
- 删除 `overlay-card`
- 删除 `stage-hud`
- 暂时保留现有 `panel`

目标是先把重复状态清掉。

### 第二步：重构页面结构

- 把 `panel` 改成顶部准备页
- `stage-shell` 改成练习页主体
- 增加结果页

目标是从双栏工作台切到 App 流程。

### 第三步：重构状态输出

- 拆分 `updateStatusUI()`
- 让不同页面各自渲染自己的状态

目标是降低耦合。

### 第四步：加入移动端沉浸态

- 增加 `is-immersive`
- 改造 `stage` 为 `100dvh`
- 处理安全区和底部 dock

目标是完成手机端全屏体验。

### 第五步：保留调试模式

- `jsonPreview`
- `hudText`
- `frameSlider`

这些能力仅在开发调试模式中显示，不进入默认界面。

## 12. 风险与注意点

### 12.1 状态迁移风险

当前多个 DOM 节点依赖同一份状态输出，删除旧结构时要同步回收对应的 `elements.*` 引用，否则容易留下空节点访问。

### 12.2 练习态信息不足风险

如果移除 `stage-hud` 后没有设计新的 `practice-dock`，用户会失去实时反馈。因此不能只做删除，必须同步补齐页面内练习信息区。

### 12.3 移动端浏览器兼容性

不同移动端浏览器对地址栏收起、底部工具栏、Fullscreen API 的支持不完全一致，因此沉浸式模式应优先依赖 CSS 视口方案。

### 12.4 调试能力丢失风险

`jsonPreview` 和 `hudText` 虽然不应出现在主界面，但对排查识别问题仍有价值，建议保留为隐藏调试功能。

## 13. 验收标准

完成改造后，至少满足以下标准：

1. 页面中不再存在 `hero`。
2. 页面中不再存在侧边栏式 `panel`。
3. 页面中不再存在 `overlay-card overlay-top collapsed`、`overlay-card overlay-bottom collapsed`、`stage-hud`。
4. 主流程变为准备页、练习页、结果页。
5. 进入全屏前，原 `panel` 的主操作均能在页面流内完成。
6. 手机端练习页支持沉浸式 `stage` 展示。
7. 调试信息默认不干扰主练习流程。

## 14. 建议的下一步开发任务

建议下一轮直接按下面的开发拆分执行：

1. 先做 DOM 和 CSS 骨架调整，移除 `hero`、`overlay-card`、`stage-hud`。
2. 再把 `panel` 拆成 `setup-view` 和 `practice-dock`。
3. 最后补 `result-view` 和移动端沉浸式样式。

这样风险最低，也最容易逐步验证。
